# OneX FDE 小班培训招生与商业化接入设计

**版本：** V1.0 设计稿  
**日期：** 2026-07-20  
**适用产品：** OneX FDE 人才网络、OneX 商业化架构  
**首期原则：** 先建立可转化、可跟进、可审计的招生框架，不绑定具体开班日期，不在首期强制上线在线支付。

## 一、设计结论

FDE 培训不是人才网络中的一张普通内容页，而是 OneX 商业化架构中的一个标准培训产品。招生页面是该产品的公开销售入口，负责解释价值、筛选申请人、收集招生申请和形成商业线索；后续合同、付款、班期、入班和交付状态通过统一商业数据接口连接。

首发产品固定为：

| 字段 | 首发值 |
|---|---|
| 产品编码 | `FDE-TRAINING-SMALL-CLASS` |
| 产品名称 | OneX FDE 小班实战培训 |
| 产品类型 | `training` |
| 招生方式 | 申请制 |
| 单班人数上限 | 10 人 |
| 班期日期 | 可为空，首期不在前台展示具体日期 |
| 首期主转化 | 提交申请并由运营沟通 |
| 在线支付 | 首期不作为上线前置条件 |
| 人才库关系 | 培训记录可作为人才申请材料，不自动产生入库、认证或徽章 |

## 二、三种接入方式与选择

### 2.1 独立报名表

招生页直接把表单发送到独立表格或单独数据库。优点是快；缺点是产品、线索、合同、付款和入班状态继续分散，后续仍需迁移。

### 2.2 商业化产品模型

将 FDE 培训作为统一产品目录中的一个产品，由商业化架构管理产品、销售方案、申请、线索、成交和入班状态。招生页只依赖稳定的商业接口，不依赖某个具体 CRM 或支付服务商。

### 2.3 直接写入人才网络

将培训申请、付款和班期直接写入人才库数据库。实现较少，但会把付费商业关系与入库、认证结论混在一起，损害认证独立性。

**确定方案：采用 2.2 商业化产品模型。**

## 三、产品边界

### 3.1 招生系统负责

- 展示培训价值、适合人群、培养路径和小班机制。
- 展示“每期最多 10 人”，不制造虚假剩余名额或虚假截止时间。
- 收集申请人的背景、经验、目标、投入条件和联系方式。
- 形成可分配、可跟进、可关闭的招生线索。
- 记录来源、运营负责人、沟通结论和后续商业状态。
- 将通过审核的申请人加入候补或指定班期。
- 为合同、付款和入班系统保留外部业务编号。

### 3.2 招生系统不负责

- 不因为报名、付款或完成培训自动授予人才库资格。
- 不因为付款自动授予 OneX FDE 认证或徽章。
- 不在匿名分析事件中保存姓名、手机号、微信号或申请正文。
- 不承诺每位申请人都能进入某一班期。
- 不在没有真实数据时显示“仅剩 N 个名额”。
- 不在首期建设退款、分账、发票和在线合同编辑器。

## 四、商业数据模型

商业化架构至少提供六个独立对象。

### 4.1 产品 `commercial_product`

描述稳定的可售产品，不绑定具体班期。

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | string | 内部唯一编号 |
| `code` | string | 固定为 `FDE-TRAINING-SMALL-CLASS` |
| `name` | string | OneX FDE 小班实战培训 |
| `type` | enum | `training` |
| `status` | enum | `draft`、`active`、`paused`、`archived` |
| `capacity_per_cohort` | integer | 固定为 10 |
| `application_mode` | enum | `review_required` |
| `public_path` | string | `/fde-training/` |
| `created_at` / `updated_at` | datetime | 服务器时间 |

### 4.2 销售方案 `commercial_offer`

描述产品在某一阶段的销售规则，可在没有具体日期时存在。

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | string | 内部唯一编号 |
| `product_id` | string | 关联产品 |
| `name` | string | 例如“FDE 小班首期申请” |
| `status` | enum | `open`、`paused`、`waitlist_only`、`closed` |
| `price_display` | string/null | 首期允许为空或显示“沟通后确认” |
| `application_open` | boolean | 是否接受新申请 |
| `cohort_id` | string/null | 未建立班期时为空 |
| `starts_at` / `ends_at` | datetime/null | 首期允许为空 |

### 4.3 班期 `training_cohort`

班期是实际交付单元。招生页首期不要求存在班期记录，但一旦创建必须遵守人数上限。

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | string | 内部唯一编号 |
| `product_id` | string | 关联 FDE 培训产品 |
| `name` | string | 内部可识别名称 |
| `capacity` | integer | 不得大于 10 |
| `starts_at` / `ends_at` | datetime/null | 可在运营确认后补充 |
| `status` | enum | `planning`、`recruiting`、`full`、`in_progress`、`completed`、`cancelled` |
| `confirmed_count` | integer | 已正式入班人数，不超过 10 |

### 4.4 招生申请 `training_application`

申请记录保存招生需要的结构化信息。

| 字段 | 类型 | 公开性 |
|---|---|---|
| `id` | string | 内部 |
| `product_id` / `offer_id` | string | 内部 |
| `name` | string | 私密 |
| `mobile` | string | 私密、加密或受控存储 |
| `wechat` | string/null | 私密 |
| `current_role` | string | 运营可见 |
| `ai_experience` | enum | 运营可见 |
| `fde_experience` | string | 运营可见 |
| `learning_goal` | string | 运营可见 |
| `time_commitment` | string | 运营可见 |
| `source` | enum | 公开测试、公众号、社群、人才页、企业推荐、直接访问、其他 |
| `source_detail` | string/null | 受限、不得保存完整外部 URL |
| `consent_version` | string | 内部 |
| `status` | enum | 见第五章 |
| `assigned_operator_id` | string/null | 内部 |
| `created_at` / `updated_at` | datetime | 服务器时间 |

### 4.5 商业机会 `commercial_opportunity`

当运营确认申请人适配并进入报价、合同或付款阶段时创建。申请未通过或仅咨询时可以不存在商业机会。

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | string | 内部唯一编号 |
| `application_id` | string | 关联招生申请 |
| `stage` | enum | `qualified`、`quoted`、`contract_pending`、`payment_pending`、`won`、`lost`、`refunded` |
| `quoted_amount` | decimal/null | 报价后填写 |
| `contract_external_id` | string/null | 外部合同系统编号 |
| `payment_external_id` | string/null | 外部支付或财务编号 |
| `lost_reason` | enum/null | 不适配、时间冲突、预算、无响应、选择其他方案、其他 |
| `owner_id` | string | 商业负责人 |

### 4.6 入班记录 `training_enrollment`

只有商业和运营条件满足后才创建正式入班记录。

| 字段 | 类型 | 规则 |
|---|---|---|
| `id` | string | 内部唯一编号 |
| `application_id` | string | 关联申请 |
| `cohort_id` | string | 关联班期 |
| `status` | enum | `reserved`、`confirmed`、`active`、`completed`、`withdrawn` |
| `seat_number` | integer | 1 至 10，班期内唯一 |
| `payment_status` | enum | `not_required`、`pending`、`paid`、`refunded` |
| `contract_status` | enum | `not_required`、`pending`、`signed`、`cancelled` |
| `completed_at` | datetime/null | 结业后填写 |

## 五、状态流转

### 5.1 招生申请状态

```text
submitted → reviewing → contacted → qualified → admitted → enrolled
                         ↘ waitlisted ↗
submitted/reviewing/contacted/qualified → rejected
任一未成交状态 → withdrawn 或 closed
```

状态说明：

- `submitted`：申请已提交。
- `reviewing`：运营正在检查背景和目标。
- `contacted`：已完成或安排首次沟通。
- `qualified`：确认适合进入商业确认阶段。
- `waitlisted`：符合条件但暂未安排班期或名额。
- `admitted`：已确认可进入指定班期，尚未正式入班。
- `enrolled`：已形成正式入班记录。
- `rejected`：当前不适合该产品，必须填写原因。
- `withdrawn`：申请人主动撤回。
- `closed`：长期无响应或运营关闭，必须填写原因。

### 5.2 商业机会状态

```text
qualified → quoted → contract_pending / payment_pending → won
任一未成交状态 → lost
won → refunded
```

商业状态不能改变人才库、认证或徽章状态。

## 六、招生页面结构

独立路由固定为 `/fde-training/`。

### 6.1 首屏

- 固定标题：`OneX FDE 小班实战培训`
- 固定人数说明：`每期最多 10 人`
- 核心主张：从会使用 AI，走向能理解企业问题、设计方案并完成可验收交付。
- 主按钮：`申请加入小班`
- 次按钮：`先测测我的 FDE 潜质`
- 首期不显示具体开班日期和未经确认的价格。

### 6.2 适合人群

- 希望转型 FDE 的开发者、产品经理、解决方案人员和企业技术骨干。
- 已经使用 AI，但缺少企业场景诊断、交付治理和验收能力的人。
- 已有项目经验，希望形成系统方法和进入人才网络申请路径的人。
- 不适合只想购买证书、只学习提示词或无法投入项目训练的人。

### 6.3 培养结果

- 识别真实企业问题、目标、证据、角色和约束。
- 设计可实施、可管控、可验收的 AI 方案。
- 使用 AI 完成可运行、可复现的成果。
- 管理权限、隐私、风险、变更、上线和交付证据。
- 完成结业考核并形成可用于人才库申请的训练记录。

### 6.4 课程能力模块

首期页面只展示稳定能力模块，不展示尚未固定的课次日期：

1. 企业问题诊断与价值定义。
2. AI 方案、知识库、智能体与工作流设计。
3. 数据、权限、安全与合规边界。
4. 项目拆解、迭代、验收与交付治理。
5. AI 实操、过程证据和成果复现。
6. 综合案例、答辩和现场条件变更。

### 6.5 小班机制

- 每期最多 10 人。
- 申请审核后入班，不采用开放式大课自动购买。
- 每位学员需要提交阶段成果并获得针对性反馈。
- 使用真实企业情境和项目任务，不只进行概念讲解。
- 允许使用 AI，但必须解释过程、取舍、失败路径和验收逻辑。

### 6.6 培训出口

```text
招生申请 → 小班训练 → 结业考核 → 训练记录
                                      ↓
                             人才库独立申请
                                      ↓
                         OneX FDE 独立综合认证
                                      ↓
                          企业需求与项目连接
```

固定边界文案：

> 完成培训或支付培训费用，不代表自动进入 OneX FDE 人才库，也不代表通过 OneX FDE 综合认证。人才入库、认证和徽章均使用独立标准、独立流程和可追溯结论。

### 6.7 申请流程

1. 申请人查看课程与小班机制。
2. 填写招生申请并同意隐私规则。
3. 系统生成申请编号和商业线索。
4. 运营完成初审并安排沟通。
5. 适配者进入候补、班期确认或商业确认。
6. 合同、付款和班期条件满足后形成入班记录。

### 6.8 常见问题

页面必须回答：

- 培训是否允许使用 AI。
- 为什么每期最多 10 人。
- 是否保证进入人才库。
- 是否保证获得认证徽章。
- 当前为什么不显示固定日期。
- 申请提交后多久联系。
- 如果暂时没有班期如何处理。
- 申请信息如何保存、撤回和删除。

## 七、表单与接口

### 7.1 公开产品读取

```http
GET /api/commercial/public/products/FDE-TRAINING-SMALL-CLASS
```

公开响应只返回产品名称、公开说明、人数上限、申请状态、价格展示文案和公开路由，不返回内部成本、合同模板、付款记录或运营人员信息。

### 7.2 招生申请提交

```http
POST /api/commercial/public/training-applications
Idempotency-Key: <client-generated-key>
Content-Type: application/json
```

请求字段固定为：

```json
{
  "product_code": "FDE-TRAINING-SMALL-CLASS",
  "offer_id": "fde-small-class-open-application",
  "name": "申请人姓名",
  "mobile": "已验证或待验证手机号",
  "wechat": "可选微信号",
  "current_role": "当前职业或岗位",
  "ai_experience": "beginner|practitioner|delivery",
  "fde_experience": "已有项目或交付经验",
  "learning_goal": "希望解决的问题和目标",
  "time_commitment": "可以投入的学习与实操时间",
  "source": "public_test|wechat_article|community|talent_page|referral|direct|other",
  "consent_version": "training-application-v1"
}
```

接口要求：

- 服务端校验字段长度、枚举、手机号格式、重复提交和限流。
- 相同幂等键返回同一申请，不重复创建线索。
- 申请成功返回公开申请编号，不返回内部运营备注。
- 若销售方案暂停，返回“暂不接受新申请”；若只开放候补，明确返回候补状态。
- 申请正文不得写入匿名分析数据库。

### 7.3 商业化适配接口

首期实现内部商业数据契约，并通过适配器连接未来 CRM、合同、支付或财务系统：

```python
class CommercialAdapter(Protocol):
    def sync_lead(self, application: dict) -> str | None: ...
    def sync_opportunity(self, opportunity: dict) -> str | None: ...
    def get_contract_status(self, external_id: str) -> str | None: ...
    def get_payment_status(self, external_id: str) -> str | None: ...
```

没有外部服务时使用 `LocalCommercialAdapter`，数据保存在独立商业数据库中；不得伪装为已连接 CRM、合同或支付平台。

## 八、后台最小能力

商业运营后台首期至少支持：

- 查看和筛选招生申请。
- 按来源、状态、背景和提交时间筛选。
- 分配运营负责人。
- 记录沟通摘要和下一步时间。
- 标记适配、不适配、候补、录取和关闭原因。
- 创建商业机会并记录报价、合同和付款外部编号。
- 创建班期并强制容量不超过 10 人。
- 将已录取申请人加入班期，阻止第 11 个确认席位。
- 导出脱敏运营清单，不导出无权限的完整联系方式。
- 对状态、负责人、商业结论、合同/付款编号和入班动作写入审计日志。

## 九、人才网络连接

招生与人才网络只通过明确记录连接：

- `training_enrollment.completed` 可生成一条“培训完成记录”。
- 培训完成记录包含产品版本、班期、完成时间和结业结论。
- 人才申请人可授权将该记录作为人才库申请材料。
- 人才库入库测试仍独立进行，不能用付款或培训完成替代。
- 综合认证仍独立进行，不能用培训成绩直接生成认证徽章。
- 对外人才主页只能在本人授权和运营审核后展示培训记录摘要。

## 十、分析与归因

匿名事件只记录漏斗，不记录个人申请内容：

| 事件 | 触发点 | 允许属性 |
|---|---|---|
| `training_page_view` | 查看招生页 | 来源、设备、语言 |
| `training_apply_start` | 开始填写 | 来源、入口位置 |
| `training_apply_submit` | 提交成功 | 来源、申请结果类型 |
| `training_apply_error` | 提交失败 | 粗粒度错误码 |

商业数据库使用申请编号连接后续运营状态；匿名分析 visitor/session ID 不回填姓名、手机号或申请正文。转化报表可按来源统计申请数、有效线索数、录取数和成交数，但不得公开个人级列表。

## 十一、安全、隐私与审计

- 申请表必须展示隐私和使用目的。
- 手机号、微信号和沟通记录只对授权运营角色开放。
- 公开页面和匿名统计不得返回或写入申请人的联系方式。
- 表单接口需要限流、幂等、输入校验、CSRF/来源校验和蜜罐字段。
- 商业状态、合同编号、付款编号和入班结果修改必须审计。
- 删除申请时，依法或因争议需要保留的商业记录应转为受限归档，并向申请人说明原因。
- 生产环境没有可用商业适配器时必须使用本地模式并明确标记，不能静默丢失线索。

## 十二、异常与边界场景

- 产品暂停：页面保留说明，只关闭新申请入口。
- 暂无班期：允许提交申请并进入候补，不展示虚构日期。
- 候补已满：停止新申请或转为下一阶段线索，由运营配置。
- 重复手机号：提示已有申请并进入验证/人工合并，不创建重复商业机会。
- 重复点击提交：幂等返回同一申请编号。
- 外部 CRM 不可用：本地申请保存成功后进入待同步队列，不阻塞用户提交。
- 合同或付款状态不同步：显示待核对，不自动确认入班。
- 班期已满：系统阻止第 11 个 `confirmed` 或 `active` 入班记录。
- 申请人撤回：停止招生跟进；已签合同或已付款的情况转人工处理。

## 十三、分阶段交付

### 13.1 商业化框架

- 产品、销售方案、申请、商业机会、班期和入班数据模型。
- FDE 培训产品种子数据。
- 公开产品接口和招生申请接口。
- 本地商业适配器、幂等、审计和待同步机制。

### 13.2 招生页面

- `/fde-training/` 完整落地页。
- 小班机制、课程能力模块、培训出口和固定边界文案。
- 招生申请表、成功页、错误和候补状态。
- 桌面端与 390 像素移动端浏览器验收。

### 13.3 商业运营

- 招生申请列表、筛选、分配和沟通记录。
- 适配、候补、录取、关闭和商业机会状态。
- 班期创建和 10 人容量控制。
- CRM、合同、付款适配器接入点。

### 13.4 人才网络衔接

- 培训完成记录。
- 本人授权后作为人才库材料引用。
- 人才库、认证和徽章独立边界的自动化测试。

## 十四、验收标准

1. `/fde-training/` 能清楚说明 FDE 小班实战培训，每期最多 10 人。
2. 页面在没有具体日期和价格时仍能完整解释产品并承接申请。
3. 申请成功后生成唯一申请编号和商业线索，重复提交不重复创建。
4. 产品暂停、开放申请和只接受候补三种状态能正确反映在页面。
5. 班期容量在数据库、接口和后台三个层面均不能超过 10 人。
6. 招生申请可以进入沟通、候补、录取、成交和入班状态，并记录原因与负责人。
7. CRM、合同和支付未接入时仍能在本地安全保存并跟进线索。
8. 接入外部商业系统时只替换适配器，不修改招生页面表单契约。
9. 姓名、手机、微信和申请正文不进入匿名分析事件。
10. 支付、培训完成、人才入库、认证和徽章五种结果在数据、文案和权限上完全分开。
11. 现有中英文公开测试、人才网络和匿名统计自动化测试无回归。
12. 桌面端与 390 像素移动端无横向溢出，表单具备键盘和基础无障碍能力。

## 十五、纳入总体实施计划

本设计作为 OneX FDE 人才网络总体计划中的独立商业化工作包，执行顺序为：

1. 建立商业产品、销售方案和申请数据契约。
2. 建立本地商业适配器和审计能力。
3. 上线 `/fde-training/` 招生框架和申请表。
4. 将公开测试结果页、“我想成为 FDE”和首页入口连接到招生页面。
5. 建立招生运营工作台、班期和最多 10 人容量约束。
6. 在真实 CRM、合同、支付服务确定后接入相应适配器。
7. 将培训完成记录按授权方式提供给人才库申请流程。

该商业化工作包与人才目录基础服务可以并行设计，但共享身份、权限、通知和审计标准。招生申请的私密商业数据不得进入公开人才库数据库或匿名统计数据库。
